# Промпти до AI та критичний аналіз результату

Згідно з вимогою курсу, для кожного завдання наведено промпт, яким користувалися,
і коментар: що в AI-відповіді **прийнято**, а що **виправлено вручну**.

> Порада для захисту/запису екрана: будьте готові пояснити кожен пункт «виправлено вручну» —
> саме критичний аналіз AI-відповіді оцінюється окремо (1 бал у кожному завданні).

---

## Завдання 1 — Docker

**Промпт:**
> Напиши multi-stage Dockerfile для Node.js (Express) застосунку на порту 3000:
> окремий етап для встановлення залежностей і збірки; у фінальному образі лише
> production-залежності; запуск від non-root користувача. Додай docker-compose.yml
> із сервісом застосунку та PostgreSQL (env-змінні, volume для даних) і .dockerignore.

**Прийнято:** загальну структуру multi-stage (builder → production), ідею окремого
копіювання `package*.json` для кешування шарів, використання образу `node:alpine`.

**Виправлено вручну:**
- AI спочатку запускав контейнер від root — додав `USER node` та `--chown=node:node`.
- Додав `npm ci --omit=dev` (замість `npm install`) для відтворюваних production-залежностей.
- У compose додав `healthcheck` для PostgreSQL та `depends_on: condition: service_healthy`,
  бо без цього app стартував раніше за готовність БД.
- Прибрав версію `version:` з compose (застаріле поле у новому Compose v2).

## Завдання 2 — JWT + RBAC

**Промпт:**
> Реалізуй на Express: POST /login (приймає email+password, повертає JWT з userId та role);
> middleware authenticate (перевіряє Bearer-токен, 401 якщо невалідний);
> middleware authorize(role) (RBAC, 403 при невідповідності); захищений GET /admin лише для admin.
> Паролі хешувати, не зберігати у відкритому вигляді.

**Прийнято:** структуру middleware, використання `jsonwebtoken` та `bcryptjs`,
розділення authenticate/authorize.

**Виправлено вручну:**
- AI зберігав паролі у відкритому вигляді в масиві — замінив на `bcrypt.hashSync`
  і `bcrypt.compare` при логіні.
- Додав однакову відповідь для «невідомий email» і «невірний пароль» (захист від user enumeration).
- Додав `expiresIn: "15m"` — AI генерував токен без терміну дії.
- Виніс `JWT_SECRET` у змінну середовища з fallback (AI хардкодив секрет у коді).
- Додав перевірку формату заголовка `Bearer <token>` (AI брав увесь заголовок як токен).

## Завдання 3 — Тестування (Jest)

**Промпт:**
> Напиши Jest unit-тести для функції calculateDiscount(price, customerType):
> покрий vip (-20%) та regular (-5%), невідомий тип (без знижки),
> граничний price === 0, і викидання помилки при від'ємній ціні через toThrow.

**Прийнято:** базову структуру `describe/test`, використання `toBe` та `toThrow`.

**Виправлено вручну / чого AI не врахував:**
- AI перевіряв `price === 0` лише для одного типу — додав перевірку для vip/regular/unknown.
- Додав окремий кейс «тип клієнта не передано» (`undefined`) — AI його пропустив.
- Уточнив `toThrow("Invalid price")` з конкретним повідомленням, а не лише фактом помилки.
- Додав тест на точність обчислень (нецілі результати), щоб переконатися у відсутності
  проблем із floating point.
